# Trade Intelligence Specification

The Trade Intelligence Store captures everything the system does — every signal,
rejection, order, fill, position update, and outcome. It is the single source of
truth for performance analysis, strategy degradation detection, and future ML training.

This is NOT just a rejected trades log. It is a complete audit of every decision
the system makes, whether acted upon or not.

See EXECUTION_SPEC.md for how rejections are emitted.
See CORE_ARCHITECTURE.md for how signals and positions are emitted.
See PUBSUB_SCHEMA.md for the Pub/Sub events that feed this store.

---

## Design Principles

1. **Everything is stored.** Signals, rejections, orders, fills, stop updates,
   position lifecycle — all of it. Nothing is dropped.
2. **Immutable append-only.** Records are never updated or deleted.
   Position closure adds a new record; it does not modify the open record.
3. **DB Writer Consumer owns all writes.** No service writes to this store directly.
   All data arrives via Pub/Sub events.
4. **Strategy Degradation Detection is a consumer, not a producer.**
   It reads from this store. It does not own the store.
5. **Enrichment is async.** The "what happened after" enrichment for rejections
   runs as a nightly batch job, not at write time.

---

## Data Flow

```
Core Binary
  ├─ events.signals       → all signals (traded + not traded)
  ├─ events.rejections    → all rejections with full context
  ├─ events.positions     → position lifecycle (open, update, close)
  ├─ events.candles       → closed candles (OHLCV)
  └─ events.risk          → risk events (kill switch, drawdown, etc.)

Executor Binary
  ├─ events.orders        → all orders submitted
  ├─ events.order_results → fill confirmations, rejections
  ├─ events.executions    → completed execution records
  └─ events.position_commands → stop updates, partial closes

           ↓ (all via Pub/Sub)

DB Writer Consumer
  └─ Writes to PostgreSQL Trade Intelligence Store

           ↓

Consumers of the store:
  ├─ Analytics API        → P&L dashboards, reporting
  ├─ Degradation Monitor  → strategy health checks
  ├─ Backtester           → compare live vs historical
  └─ Future: ML pipeline  → supervised learning on outcomes
```

## Database Schema

### 1. signals

Every signal generated by the strategy engine.
One record per signal, regardless of whether it was traded.

```sql
CREATE TABLE signals (
  signal_id           UUID PRIMARY KEY,
  timestamp           TIMESTAMPTZ NOT NULL,
  symbol              VARCHAR(30) NOT NULL,
  exchange            VARCHAR(10) NOT NULL,
  segment             VARCHAR(10) NOT NULL,
  strategy_id         VARCHAR(100) NOT NULL,
  strategy_version    VARCHAR(20) NOT NULL,
  direction           VARCHAR(10) NOT NULL,    -- BUY | SELL
  timeframe           VARCHAR(10) NOT NULL,
  bar_close_time      TIMESTAMPTZ NOT NULL,
  entry_mode          VARCHAR(10) NOT NULL,    -- AND | SCORE

  -- Condition snapshot at signal time
  conditions_met      TEXT[] NOT NULL,
  conditions_failed   TEXT[],
  score               DECIMAL(6,4),            -- null if AND mode
  signal_strength     DECIMAL(6,4) NOT NULL,   -- 1.0 for AND, 0-1 for SCORE

  -- What happened with this signal
  acted_upon          BOOLEAN NOT NULL,
  order_id            UUID,                    -- null if not traded
  rejection_id        UUID,                    -- null if traded

  -- Market context at signal time
  india_vix           DECIMAL(8,4),
  regime              VARCHAR(20),
  bar_atr             DECIMAL(12,4)
);

SELECT create_hypertable('signals', 'timestamp');
CREATE INDEX ON signals (strategy_id, timestamp DESC);
CREATE INDEX ON signals (symbol, timestamp DESC);
CREATE INDEX ON signals (acted_upon, timestamp DESC);
```

### 2. rejections

Every signal that was generated but not traded, with full context at each stage.

```sql
CREATE TABLE rejections (
  rejection_id           UUID PRIMARY KEY,
  timestamp              TIMESTAMPTZ NOT NULL,
  symbol                 VARCHAR(30) NOT NULL,
  strategy_id            VARCHAR(100) NOT NULL,
  strategy_version       VARCHAR(20) NOT NULL,
  signal_id              UUID REFERENCES signals(signal_id),

  -- Where in the pipeline it was rejected
  rejection_stage        SMALLINT NOT NULL,    -- 1-6
  rejection_reason       VARCHAR(100) NOT NULL,

  -- Signal details
  direction              VARCHAR(10) NOT NULL,
  timeframe              VARCHAR(10) NOT NULL,
  bar_close_time         TIMESTAMPTZ NOT NULL,

  -- Full scoring context
  signal_strength        DECIMAL(6,4),
  win_rate_component     DECIMAL(6,4),
  allocator_weight_component DECIMAL(6,4),
  regime_match_component DECIMAL(6,4),
  composite_score        DECIMAL(6,4),
  score_threshold        DECIMAL(6,4),
  passed_threshold       BOOLEAN,
  fallbacks_used         TEXT[],

  -- R:R context (null if rejected before R:R stage)
  rr_entry_price         DECIMAL(12,4),
  rr_stop_price          DECIMAL(12,4),
  rr_target_price        DECIMAL(12,4),
  rr_ratio               DECIMAL(6,4),
  rr_min_required        DECIMAL(6,4),

  -- Risk context at rejection time
  daily_pnl_at_rejection DECIMAL(12,2),
  daily_loss_limit       DECIMAL(12,2),
  open_positions_count   SMALLINT,
  available_margin_inr   DECIMAL(14,2),
  kill_switch_level      SMALLINT,

  -- Market context
  india_vix              DECIMAL(8,4),
  regime                 VARCHAR(20),
  session                VARCHAR(20),

  -- Enriched nightly (what actually happened after)
  price_1bar_after       DECIMAL(12,4),
  price_5bars_after      DECIMAL(12,4),
  price_20bars_after     DECIMAL(12,4),
  would_have_profited    BOOLEAN,
  max_favourable_excursion_pct DECIMAL(8,4),
  max_adverse_excursion_pct    DECIMAL(8,4),
  enriched_at            TIMESTAMPTZ
);

SELECT create_hypertable('rejections', 'timestamp');
CREATE INDEX ON rejections (strategy_id, timestamp DESC);
CREATE INDEX ON rejections (rejection_stage, rejection_reason);
CREATE INDEX ON rejections (symbol, timestamp DESC);
CREATE INDEX ON rejections (would_have_profited, rejection_stage);
```

### 4. positions

Complete position lifecycle. Open record updated when closed.

```sql
CREATE TABLE positions (
  position_id          UUID PRIMARY KEY,
  symbol               VARCHAR(30) NOT NULL,
  exchange             VARCHAR(10) NOT NULL,
  segment              VARCHAR(10) NOT NULL,
  strategy_id          VARCHAR(100) NOT NULL,
  strategy_version     VARCHAR(20) NOT NULL,

  -- Entry
  entry_order_id       UUID REFERENCES orders(order_id),
  direction            VARCHAR(10) NOT NULL,
  lots                 SMALLINT NOT NULL,
  quantity             INTEGER NOT NULL,
  entry_price          DECIMAL(12,4) NOT NULL,
  entry_time           TIMESTAMPTZ NOT NULL,
  execution_mode       VARCHAR(10) NOT NULL,

  -- Stop / Target at entry
  initial_stop_price   DECIMAL(12,4) NOT NULL,
  initial_target_price DECIMAL(12,4),
  initial_rr_ratio     DECIMAL(6,4),
  stop_type            VARCHAR(20),           -- fixed_pct | atr_based

  -- Current state (updated on each stop modification)
  current_stop_price   DECIMAL(12,4),
  stop_modification_count SMALLINT DEFAULT 0,

  -- Exit
  exit_order_id        UUID REFERENCES orders(order_id),
  exit_price           DECIMAL(12,4),
  exit_time            TIMESTAMPTZ,
  exit_reason          VARCHAR(50),           -- stop_loss | take_profit | time_exit |
                                              -- signal_exit | forced | emergency | trailing_stop

  -- P&L
  gross_pnl_inr        DECIMAL(12,2),
  brokerage_inr        DECIMAL(10,2),
  stt_inr              DECIMAL(10,2),
  net_pnl_inr          DECIMAL(12,2),
  net_pnl_pct          DECIMAL(8,4),
  r_multiple           DECIMAL(6,4),          -- net_pnl / initial_risk (how many R did we make?)

  -- Hold duration
  hold_bars            SMALLINT,
  hold_minutes         INTEGER,

  -- Max excursions during trade
  max_favourable_excursion_pct DECIMAL(8,4),  -- best unrealised profit during trade
  max_adverse_excursion_pct    DECIMAL(8,4),  -- worst drawdown during trade

  -- Status
  status               VARCHAR(10) NOT NULL   -- OPEN | CLOSED
);

CREATE INDEX ON positions (strategy_id, entry_time DESC);
CREATE INDEX ON positions (symbol, entry_time DESC);
CREATE INDEX ON positions (status, entry_time DESC);
CREATE INDEX ON positions (execution_mode, entry_time DESC);
```

### 6. strategy_performance_snapshots

Daily snapshot of each strategy's live performance vs backtest baseline.
Used by Strategy Degradation Detection.

```sql
CREATE TABLE strategy_performance_snapshots (
  snapshot_id          UUID PRIMARY KEY,
  snapshot_date        DATE NOT NULL,
  strategy_id          VARCHAR(100) NOT NULL,
  strategy_version     VARCHAR(20) NOT NULL,
  execution_mode       VARCHAR(10) NOT NULL,   -- paper | live

  -- Rolling window metrics (last 30 days or last 50 trades, whichever is less)
  window_trades        INTEGER NOT NULL,
  window_days          INTEGER NOT NULL,

  -- Live metrics
  live_win_rate        DECIMAL(6,4),
  live_avg_pnl_inr     DECIMAL(12,2),
  live_sharpe          DECIMAL(8,4),
  live_avg_rr_achieved DECIMAL(6,4),           -- avg R multiple achieved
  live_profit_factor   DECIMAL(8,4),
  live_max_dd_pct      DECIMAL(8,4),

  -- Backtest baseline (from strategy.backtest_results)
  backtest_win_rate    DECIMAL(6,4),
  backtest_avg_pnl_inr DECIMAL(12,2),
  backtest_sharpe      DECIMAL(8,4),
  backtest_profit_factor DECIMAL(8,4),

  -- Divergence metrics
  win_rate_divergence_pct     DECIMAL(8,4),   -- (live - backtest) / backtest × 100
  sharpe_divergence_pct       DECIMAL(8,4),
  avg_pnl_divergence_pct      DECIMAL(8,4),

  -- Degradation status
  degradation_status   VARCHAR(20),           -- NORMAL | WARNING | CRITICAL | DEGRADED
  degradation_triggered BOOLEAN DEFAULT FALSE,
  triggered_at         TIMESTAMPTZ
);

CREATE UNIQUE INDEX ON strategy_performance_snapshots (strategy_id, snapshot_date, execution_mode);
CREATE INDEX ON strategy_performance_snapshots (degradation_status, snapshot_date DESC);
```

## Strategy Degradation Detection

Runs nightly as part of EOD jobs (Cloud Scheduler → Cloud Run).
Reads from `strategy_performance_snapshots` and `positions` tables.
Publishes to `events.risk` if degradation detected.

### Detection Algorithm

```
For each active strategy (paper + live):

  1. Gather rolling window data
     Look back: last 30 calendar days OR last 50 completed trades (whichever has more)
     Minimum required: 10 trades (skip if insufficient data)

  2. Calculate live metrics
     win_rate         = count(net_pnl_inr > 0) / count(*)
     avg_pnl_inr      = avg(net_pnl_inr)
     sharpe           = mean(daily_returns) / stddev(daily_returns) × sqrt(252)
     profit_factor    = sum(profitable trades) / abs(sum(losing trades))

  3. Compare to backtest baseline
     win_rate_div   = (live_win_rate - backtest_win_rate) / backtest_win_rate × 100
     sharpe_div     = (live_sharpe - backtest_sharpe) / abs(backtest_sharpe) × 100
     avg_pnl_div    = (live_avg_pnl - backtest_avg_pnl) / abs(backtest_avg_pnl) × 100

  4. Apply degradation thresholds

     NORMAL:    all divergences within tolerance
     WARNING:   any one metric diverges > 20% for 5+ consecutive days
     CRITICAL:  any one metric diverges > 35% for 3+ consecutive days
     DEGRADED:  any one metric diverges > 50%,
                OR: 3+ consecutive metrics at WARNING simultaneously

  5. On DEGRADED status:
     → Publish events.risk: STRATEGY_DEGRADATION {strategy_id, metrics}
     → Core Risk Monitor reads this event
     → Sets strategy.status = STRATEGY_DD_WARNING in Redis
     → If already WARNING: escalates to STRATEGY_HALTED
     → Operator receives CRITICAL alert
```

### Degradation Thresholds

```
Metric                  WARNING     CRITICAL    DEGRADED
────────────────────────────────────────────────────────
Win Rate divergence      -20%        -35%        -50%
Sharpe divergence        -20%        -35%        -50%
Avg P&L divergence       -20%        -35%        -50%
Profit Factor < 1.0      3 days      7 days      sustained

Consecutive days at WARNING before escalating to CRITICAL: 5
Consecutive days at CRITICAL before escalating to DEGRADED: 3

Volatility adjustment:
  If India VIX > 20: thresholds widened by 30%
  (High volatility markets naturally cause divergence from backtests)
  WARNING becomes -26%, CRITICAL becomes -45.5%, etc.
```

## Nightly Enrichment Job (What Happened After)

Runs nightly for all rejections from the current day.
Tags each rejection with what price actually did afterward.

```
For each rejection from today (rejections table):

  symbol     = rejection.symbol
  timeframe  = rejection.timeframe
  bar_close  = rejection.bar_close_time
  direction  = rejection.direction

  Fetch candles from TimescaleDB:
    1 bar after  bar_close_time → price_1bar_after
    5 bars after bar_close_time → price_5bars_after
   20 bars after bar_close_time → price_20bars_after

  Calculate:
    If direction = BUY:
      would_have_profited = price_5bars_after > rejection.rr_entry_price
      max_fav = max(high prices in next 20 bars - entry_price) / entry_price × 100
      max_adv = min(low prices in next 20 bars - entry_price) / entry_price × 100

    If direction = SELL:
      would_have_profited = price_5bars_after < rejection.rr_entry_price
      (reversed calculation)

  Update rejections table: enriched_at, would_have_profited,
  max_favourable_excursion_pct, max_adverse_excursion_pct
```

### Using Enrichment Data

```sql
-- Were our risk check rejections actually good trades we missed?
SELECT
  rejection_reason,
  COUNT(*) as total,
  SUM(CASE WHEN would_have_profited THEN 1 ELSE 0 END) as would_have_won,
  AVG(max_favourable_excursion_pct) as avg_profit_missed
FROM rejections
WHERE rejection_stage = 3              -- risk check
  AND timestamp > NOW() - INTERVAL '30 days'
  AND enriched_at IS NOT NULL
GROUP BY rejection_reason
ORDER BY total DESC;

-- Were our R:R rejections leaving money on the table?
SELECT
  COUNT(*) as rr_rejections,
  AVG(rr_ratio) as avg_rr_at_rejection,
  SUM(CASE WHEN would_have_profited THEN 1 ELSE 0 END)::FLOAT / COUNT(*) as win_rate_if_taken,
  AVG(max_favourable_excursion_pct) as avg_profit_missed_pct
FROM rejections
WHERE rejection_reason = 'RR_BELOW_THRESHOLD'
  AND timestamp > NOW() - INTERVAL '90 days';

-- Strategy health: live vs backtest by strategy
SELECT
  strategy_id,
  degradation_status,
  live_win_rate,
  backtest_win_rate,
  win_rate_divergence_pct,
  live_sharpe,
  backtest_sharpe
FROM strategy_performance_snapshots
WHERE snapshot_date = CURRENT_DATE
ORDER BY win_rate_divergence_pct ASC;
```

## Data Retention Policy

```
signals:           2 years   (active analysis window)
rejections:        2 years   (with enrichment data)
orders:            5 years   (regulatory compliance)
positions:         5 years   (tax records, regulatory)
stop_updates:      1 year    (trailing stop audit)
strategy_snapshots: 5 years  (performance history)

After retention period:
  → Archive to Parquet on Cloud Storage
  → Delete from PostgreSQL
  → Parquet kept indefinitely (cold storage)

Archive job: runs 1st of each month (Cloud Scheduler)
```
